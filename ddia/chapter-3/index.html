DDIA · Ch.3

Storage & Retrieval

Apr 02, 2023 · 14 min read · DDIA

Chapter 3 answers the question every backend engineer should be able to answer in an interview: "How does a database actually store and retrieve data?" Kleppmann walks through two fundamentally different storage engine families, explains how indexes work, and then pivots to the analytical world of data warehouses and column stores.

⚡ Quick Takeaways
  • LSM-tree favors writes — sequential appends to memtable then flushed SSTables; Bloom filters skip unnecessary disk reads. Used by Cassandra, RocksDB, HBase.
  • B-tree favors reads — one path root-to-leaf; in-place page updates with WAL for crash recovery. Used by PostgreSQL, MySQL/InnoDB, SQLite.
  • Write-ahead log (WAL) — both B-trees and LSM-trees use a WAL for durability; B-tree WAL is for crash recovery of in-place writes; LSM-tree WAL recovers the unflushed memtable.
  • Clustered vs non-clustered indexes — clustered stores row data inside the index leaf (InnoDB primary key); non-clustered stores a pointer to the heap file; a covering index includes extra columns to avoid the heap lookup entirely.
  • OLTP vs OLAP are fundamentally different access patterns — OLTP: random small point queries; OLAP: full sequential scans over millions of rows. Run them on separate systems.
  • Column-oriented storage — stores all values for one column together; reads only needed columns; compresses aggressively (bitmap, RLE, dictionary); enables CPU SIMD vectorized execution.
tldr

LSM-trees (log-structured) favor writes; B-trees favor reads. Both use indexes to make queries fast, with different tradeoffs. OLTP databases optimize for random-access point queries; OLAP warehouses optimize for sequential full-column scans. Column-oriented storage is the key insight that makes analytical queries fast.

Original Notes

The handwritten study notes this chapter summary is based on:

Log-Structured Storage Engines

The simplest database is an append-only log: write goes to the end of a file, read scans from the beginning. Appends are extremely fast (sequential I/O), but reads are O(n). Indexes solve the read problem at the cost of write overhead.

Hash Index

Bitcask (the default storage engine in Riak) keeps an in-memory hash map from every key to the byte offset of its latest value in the log file. Writes append to the active segment; an in-memory pointer is updated. Reads do one disk seek to the offset. This works when the entire key set fits in RAM (common for high-write-rate, moderate-key-count workloads like URL shorteners). Compaction merges old segments, discarding overwritten values to reclaim disk space. Crash recovery restores the hash map from a snapshot.

SSTables and LSM-Trees

Sorted String Tables (SSTables) impose a constraint: within each segment file, keys are sorted. This enables efficient merging (like merge sort), sparse in-memory indexes (one entry every few KB rather than one per key), and range queries. The Log-Structured Merge-Tree (LSM-Tree), described by Patrick O'Neil et al. in 1996 and popularized by LevelDB, RocksDB, Cassandra, and HBase, is built on SSTables:

  1. Writes go to an in-memory balanced BST (memtable), e.g. a red-black tree.
  2. When the memtable exceeds a threshold (~a few MB), it is flushed to disk as a new SSTable segment — already sorted.
  3. Reads check the memtable first, then the most recent on-disk SSTable, then older ones in order. A Bloom filter per SSTable quickly answers "key definitely not here" to avoid unnecessary disk reads.
  4. Background compaction merges SSTable levels. Leveled compaction (LevelDB/RocksDB) keeps total size per level bounded; size-tiered compaction (Cassandra default) groups similar-size SSTables.

A crash before a memtable flush is recovered via a separate write-ahead log (WAL) that is discarded once the flush completes.

B-Tree Storage Engines

B-trees are the standard index structure for relational databases (PostgreSQL, MySQL/InnoDB, SQLite). Unlike LSM-trees that break data into variable-length segments, B-trees break data into fixed-size pages (traditionally 4 KB, matching disk block size). The tree is balanced: leaf pages contain values (or references to values), internal pages contain keys and pointers to child pages. A write updates the page in place, then writes a Write-Ahead Log (WAL) entry for crash recovery.

Write amplificationLower (sequential, batched)Higher (random page updates + WAL)

Other Index Structures

Secondary Indexes

The primary index maps a primary key to a row. A secondary index maps some other column's value to one or more row locations. Both B-trees and LSM-trees can serve secondary indexes. The key difference: primary key values are unique, secondary index values are not — each entry may point to a list of matching rows.

Clustered vs Non-Clustered Indexes

In a clustered index (InnoDB's primary key, for example), the actual row data is stored inside the index leaf page. Lookups by primary key are extremely fast — one disk access fetches both index and data. In a non-clustered (heap file) index, the index stores a pointer to the row's location in a separate heap file. A covering index is a middle ground: the index includes extra columns so that the query can be answered entirely from the index without touching the heap.

In-Memory Databases

Durability requires writing to disk eventually, but the primary bottleneck for in-memory databases is not RAM speed — it is the overhead of encoding data structures into disk-friendly formats. VoltDB, MemSQL (now SingleStore), and Redis all keep the authoritative dataset in RAM (with persistence options). Anti-caching extends the dataset beyond RAM by evicting LRU pages to disk, similar to virtual memory but managed at the database level.

OLTP vs OLAP

Online Transaction Processing (OLTP) databases serve application end-users: high volume, small random-access queries touching a few rows at a time. Online Analytical Processing (OLAP) serves data analysts: low volume, huge scans aggregating millions of rows to answer business questions. The access patterns are so different that organizations run separate systems for each.

Data Warehouses and Star Schema

A data warehouse is a separate read-optimized database loaded from OLTP systems via ETL (Extract, Transform, Load). The dominant schema is the star schema: a central fact table (each row is an event — a sale, a click, an order) connected to multiple dimension tables (product, customer, date, store). The name comes from the diagram: fact table in the center, dimension tables radiating out like star points. A snowflake schema further normalizes dimension tables into sub-dimensions.

Column-Oriented Storage

Analytical queries typically touch only a few columns out of a table with hundreds. Row-oriented storage must load the entire row to extract one column. Column-oriented storage (used by Redshift, Parquet, Clickhouse, Vertica) stores all values for one column together, so a query touching 3 columns out of 100 reads only 3% of the data.

Column Compression

Column values are often repetitive (e.g., a product_id column with only a few thousand distinct values across billions of rows). This compresses extremely well. Common techniques:

  • Bitmap encoding — one bit per row per distinct value. For low-cardinality columns, store a sparse bitmap (run-length encoded).
  • Run-length encoding — consecutive identical values compressed to (value, count). Effective when data is sorted by the column.
  • Dictionary encoding — map distinct values to small integers, store integers. Used by Parquet and ORC.

Compression also enables vectorized processing: the query engine operates on compressed column chunks using CPU SIMD instructions without decompressing first, dramatically reducing CPU cycles per row processed.

Sort Order in Column Stores

Rows can be sorted by one or more columns (like a clustered index). Sorting enables better run-length compression for the first sort key and speeds up range queries and GROUP BY on that key. Vertica even stores multiple sort orders on different replicas — different queries pick the replica whose sort order helps them most.

takeaway

For OLTP writes: LSM-tree beats B-tree on write throughput; B-tree beats LSM on read latency and predictability. For analytics: column-oriented storage is a near-universal win — less I/O, better compression, and CPU-friendly vectorized execution. Know that a data warehouse's star schema (fact + dimension tables) is the foundation for every SQL analytics query you will ever see in a BI tool or interview whiteboard.

🎯 interview hot-takes

LSM-tree vs B-tree — when do you pick each? LSM-tree for write-heavy workloads (Cassandra, RocksDB): sequential append + compaction give higher write throughput, better compression, but slower and less predictable reads. B-tree for read-heavy or mixed workloads (PostgreSQL, MySQL): one root-to-leaf path for fast predictable reads, but random in-place page writes have higher write amplification.
Why does column-oriented storage make analytics faster? An OLAP query typically touches 3–5 columns out of hundreds. Row storage must load entire rows to extract those columns. Column storage reads only the needed columns — a fraction of total I/O. Combined with high compression (repetitive column values) and CPU SIMD vectorized execution on compressed chunks, it dramatically reduces both I/O and CPU cycles.
What is a Bloom filter and why do LSM-trees use one? A Bloom filter is a probabilistic data structure that definitively answers "key definitely not here" with no false negatives. LSM-trees attach one per SSTable so reads can skip disk seeks to SSTables that cannot contain the key — avoiding expensive I/O for non-existent keys.
