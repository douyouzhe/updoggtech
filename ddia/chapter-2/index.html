DDIA · Ch.2

Data Models & Query Languages

Apr 02, 2023 · 12 min read · DDIA

Data models are the most consequential decision in any system. They shape not only how software is written, but how we think about the problem at all. Chapter 2 surveys the three dominant families — relational, document, and graph — and examines the query languages that each naturally pairs with.

tldr

Relational wins for many-to-many joins and strong schema guarantees. Document wins for self-contained hierarchical data and schema flexibility. Graph wins when relationships are the data. Know the tradeoffs; most production systems mix all three.

Original Notes

The handwritten study notes this chapter summary is based on:

Relational Model vs Document Model

Edgar Codd proposed the relational model in 1970: data is organized into relations (tables), each a collection of tuples (rows). For decades this dominated, and SQL remains the lingua franca of data. NoSQL emerged in the 2010s with a different set of motivations: greater scalability (including writes), schema flexibility for rapidly evolving data, and a better fit for certain data structures that are awkward to express in tables.

Object-Relational Impedance Mismatch

Most application code today is written in object-oriented languages. Data lives in tables. The translation between the two — handled by ORMs like Hibernate or ActiveRecord — is the impedance mismatch. It is not just an inconvenience; it forces developers to mentally maintain two representations of the same data and can hide performance problems (the N+1 query problem being the classic example).

For a résumé (LinkedIn profile) example: a user has one name, but multiple jobs and multiple education entries. In SQL you need separate tables with foreign keys for positions and education, then a JOIN to reconstruct the profile. In a document store (MongoDB, CouchDB) you store the entire profile as one JSON document — the data locality is natural and there is no JOIN needed.

One-to-Many, Many-to-One, Many-to-Many

One-to-many relationships (user → positions) are the document model's strong suit: embed the child documents inside the parent. Many-to-one relationships (many users share the same city name, stored as an ID referencing a cities table) are better served by normalization in a relational database. When you normalize, you eliminate redundancy but you also need joins. Document databases have weak join support, so many-to-one relationships require application-level joins — moving join logic from the database into application code.

Many-to-many relationships (a book has many authors, an author has many books) push the document model hardest. Early network-model databases like IDS and CODASYL attempted to solve this with manual access-path traversal; the relational model replaced that with the query optimizer doing the path selection automatically — a major usability win.

Schema-on-Read vs Schema-on-Write

PropertySchema-on-Write (Relational)Schema-on-Read (Document)
Schema enforcementAt write time by the DBAt read time by application code
Adding a new fieldALTER TABLE migration (can lock table)Just start writing new documents; old ones return null
Heterogeneous dataHard — every row must fit the schemaEasy — different documents can have different fields

Schema-on-read is sometimes called "schemaless," but this is misleading — the schema exists implicitly in application code. The analogy Kleppmann uses: schema-on-write is like static (compile-time) type checking, schema-on-read is like dynamic (runtime) type checking.

Data Locality

Document databases often store a document as a single contiguous encoded blob (JSON, BSON, XML). If your application frequently needs the entire document, this data locality advantage reduces I/O. However, if you only need part of a document, you still load the whole blob. And updates that grow a document may require rewriting it entirely. Locality therefore helps most for documents that are read as a whole and updated infrequently.

Query Languages

Declarative vs Imperative

SQL is declarative: you describe what you want, not how to get it. The query optimizer decides whether to use an index, which join algorithm to pick, and whether to parallelize the execution. This gives the database engine freedom to improve performance without changing the query. CSS selectors and XPath are also declarative.

MapReduce is a programming model (not purely declarative, not purely imperative) for processing large datasets across many machines. The user writes map and reduce functions in a general-purpose language, giving full expressive power, but losing the optimizer's ability to rewrite the query. MongoDB added an aggregation pipeline — a more declarative alternative to its JavaScript-based MapReduce.

Graph Data Model

When many-to-many relationships are pervasive and the data is highly connected, a graph model is the most natural representation. A property graph (used by Neo4j) consists of vertices and edges, each carrying a map of properties. Edges have a direction (head vertex → tail vertex) and a label. Crucially, there is no schema: any vertex can have an edge to any other vertex.

Cypher Query Language

Cypher is Neo4j's declarative query language for property graphs. A pattern like (person)-[:BORN_IN]->()-[:WITHIN*0..]->(us:Location {name:'United States'}) finds all people born anywhere in the USA, traversing any number of WITHIN edges upward through a location hierarchy. Writing this in SQL requires recursive common table expressions (WITH RECURSIVE) — possible but verbose.

Triple-Stores and SPARQL

A triple-store models everything as (subject, predicate, object) triples, e.g. (Jim, likes, bananas). RDF (Resource Description Framework) is a common format, and SPARQL is the query language for it. This model closely mirrors how knowledge graphs and semantic web data are represented. Datalog is an older precursor that underpins Datomic and is also worth knowing.

takeaway

No single data model wins universally. Relational normalizes and joins; document embeds and shards by document; graph traverses relationships efficiently. In interviews: lead with the data's relationship structure. One-to-many self-contained data → document. Complex many-to-many → relational or graph. Schema flexibility needs → document (schema-on-read). Always mention the impedance mismatch and the cost of application-level joins when document stores lack native join support.
