Sharding

Apr 02, 2023 · 10 min read · Tech Stacks

Sharding is a technique for horizontally partitioning a database across multiple independent nodes, each called a shard, so that the dataset and the write load are distributed rather than concentrated on a single machine. When a single database server can no longer handle the volume of data or query throughput — and vertical scaling (bigger CPU, more RAM) has hit its ceiling — sharding is the next step.

tldr

Sharding splits data across multiple database nodes by a shard key. Hash-based sharding gives even distribution; range-based sharding enables efficient range scans. Consistent hashing minimizes data movement when resharding. The cost is complexity: cross-shard queries, joins, and transactions require significant engineering effort.

Vertical vs. Horizontal Partitioning

DimensionVertical PartitioningHorizontal Partitioning (Sharding)
What is splitColumns — different tables/columns on different servers (functional decomposition)Rows — the same table spread across multiple nodes
Scale limitBounded by the largest single-table row countScales horizontally with node count
Use caseSeparating hot columns, microservice decompositionVery large tables that outgrow a single node

Sharding Strategies

Hash-Based Sharding

Apply a hash function to the shard key and compute hash(key) % num_shards to determine which shard holds a given row. This distributes data evenly across shards regardless of the key's natural ordering.

  • Pro: Uniform distribution; no single shard becomes disproportionately large.
  • Con: Range queries require querying all shards. Adding or removing a shard changes num_shards and invalidates all mappings, requiring a full data migration.

Range-Based Sharding

Partition rows by contiguous ranges of the shard key — e.g., user IDs 1–1,000,000 on shard 1, 1,000,001–2,000,000 on shard 2. Often used when the key has a natural ordering (timestamps, sequential IDs).

  • Pro: Range queries can target a single shard. Easy to reason about data placement.
  • Con: Hotspot risk — if recent data is the most active, the highest-range shard bears all the write load (a common problem with time-series data and sequential IDs).

Directory-Based Sharding

A lookup table (the "directory") maps each key to its shard. The application queries the directory on every request to locate the right shard.

  • Pro: Flexible — any key can be remapped to any shard without changing the sharding logic.
  • Con: The directory is a single point of failure and a performance bottleneck; it must be highly available and low-latency.

Consistent Hashing

Standard hash-based sharding forces a full data reshuffle when nodes are added or removed. Consistent hashing solves this by mapping both keys and nodes onto a logical ring. When a node is added, it only claims the keyspace between itself and its predecessor — all other assignments remain stable. When a node is removed, its keys migrate only to its successor. This means roughly k/n keys move when adding one node to an n-node cluster (where k is total key count), rather than remapping nearly everything.

Most production consistent hashing implementations use virtual nodes: each physical node owns multiple positions on the ring, which smooths out uneven distribution that can arise with small cluster sizes.

python 
import hashlib
from sortedcontainers import SortedDict

class ConsistentHashRing:
    def __init__(self, virtual_nodes=150):
        self.ring = SortedDict()
        self.vnodes = virtual_nodes

    def add_node(self, node):
        for i in range(self.vnodes):
            h = int(hashlib.md5(f"{node}-{i}".encode()).hexdigest(), 16)
            self.ring[h] = node   # place virtual node on ring

    def get_node(self, key):
        h = int(hashlib.md5(key.encode()).hexdigest(), 16)
        idx = self.ring.bisect_right(h) % len(self.ring)
        return self.ring.peekitem(idx)[1]   # nearest clockwise node

Choosing a Shard Key

The shard key is the most consequential design decision in a sharded system. A bad key creates hotspots that nullify the benefits of sharding. Good shard key properties:

  • High cardinality — enough distinct values to spread data across all shards.
  • Even write distribution — avoid monotonically increasing keys (auto-increment IDs, timestamps) with hash-based sharding; they naturally create a write hotspot on the "current" shard.
  • Aligned with access patterns — queries that include the shard key can be routed to a single shard; queries without it must fan out to all shards (scatter-gather).

Challenges & Tradeoffs

  • Cross-shard joins — SQL joins across shards require the application layer to fetch data from multiple shards and merge it in memory. Expensive and complex.
  • Distributed transactions — ACID transactions that span multiple shards need two-phase commit (2PC) or saga patterns, both of which add latency and failure surface area.
  • Rebalancing — as data grows unevenly, shards may need to be split or merged. With consistent hashing this is less disruptive, but still a significant operational event.
  • Schema changes — DDL changes (adding a column) must be applied to every shard, usually with online schema change tooling (pt-online-schema-change, gh-ost).
takeaway

Shard only when you have exhausted vertical scaling and read replicas. The added operational complexity is real. When you do shard, invest heavily in shard key design — a well-chosen key keeps queries single-shard and writes balanced. Use consistent hashing to minimize data movement during topology changes, and plan your cross-shard query strategy before the schema is set in stone.
