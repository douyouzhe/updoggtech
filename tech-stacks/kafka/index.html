Apache Kafka

Mar 31, 2023 · 8 min read · Tech Stacks

Kafka is a durable, distributed commit log — not just a message queue.

Producers append messages to topics; consumers pull from them at their own pace. Thousands of companies run it for data pipelines, streaming analytics, and async integration.

tldr

A partitioned, replicated log. Partitions give scale + per-key ordering, consumer groups give parallel reads, and acks + ISR + idempotence let you dial durability from fire-and-forget to exactly-once.

What it's good for

  • Buffer traffic spikes — soak up bursts (lunch rush, flash sales) so downstream isn't crushed.
  • Decouple services — sources don't know about consumers; MirrorMaker moves data across zones.
  • Go async — push non-critical work off the request path.

Pub/Sub model

Messages live in topics and persist after being read.

  • Producers write to topics; many consumers read the same topic independently.
  • Unlike a queue, a message isn't deleted on consume — Kafka manages retention, so consumers read at their own offset.

Producer: how a message is sent

Serialize → pick partition → batch → send & wait for ack.

  • Serialize — message → byte stream (Protobuf / Avro via Schema Registry).
  • Partition — hash(key) % partitions decides where it lands.
  • Batch — buffered in the RecordAccumulator; flushed when batch.size fills or linger.ms elapses.
  • Send — the Sender ships it and waits for the acks reply.

Consumer: pull, not push

Consumers fetch on their own schedule; one partition belongs to one member per group.

  • Pull model → more decoupling than push.
  • Many independent consumers can read the same topic.
  • Inside one consumer group, a partition is owned by exactly one member — the group is a single logical subscriber.

Partitions & replicas

Partitions give parallelism; replicas give durability.

  • Partition = append-only log, segmented at 1 GB, sparse index every 4 KB.
  • Retention (log.retention.*, default 7 days) deletes or compacts old data.
  • Replication uses leader/follower to survive broker failures.

Durability: the acks knob

acks trades throughput for durability — pick your point.

SettingBehavior
acks=0Done the instant it's sent. Fastest, least safe.
acks=1Done when the leader has it.
acks=allDone when all in-sync replicas (ISR) have it. Safest. Pair with min.insync.replicas>1.

Delivery semantics

  • At-most-once (acks=0) — no dupes, may lose messages.
  • At-least-once (acks=-1) — never lost, may duplicate.
  • Exactly-once — at-least-once + idempotence: each producer has a PID + per-message sequence; brokers drop duplicate <PID, seq> (enable.idempotence=true). Transactions wrap sends in beginTransaction() / commitTransaction().

Replicas & ISR

  • Producers write only to the leader; followers fetch from it.
  • ISR = in-sync followers. Lag past replica.lag.time.max.ms → kicked to OSR.
  • Leader dies → new leader elected from the ISR.

Why it's fast

  • Sequential writes — append-only beats random access.
  • Sparse index — .index files speed up reads.
  • Page cache — leans on the OS cache, not a custom one.
  • Zero-copy — read buffer → socket buffer, skipping user/kernel copies (lost under SSL).

ZooKeeper's job

Coordinates the cluster: who's alive, who's controller, what topics exist.

  • Controller election + live broker list (cluster membership).
  • Topic config, ACLs, and client quotas.

Try it: the CLI

Create a topic (3 replicas, 2 partitions), then produce & consume.

bash 
kafka-topics --bootstrap-server localhost:9092 \
  --topic test1 --create \
  --replication-factor 3 --partitions 2

# produce keyed messages
kafka-console-producer --topic test1 --broker-list localhost:9092 \
  --property parse.key=true --property key.separator=:

# consume from the start
kafka-console-consumer --topic test1 --bootstrap-server localhost:9092 \
  --property print.key=true --from-beginning
takeaway

Think "replicated log," not "queue." Partitions = scale + ordering-per-key, consumer groups = parallel reads, acks + ISR + idempotence = tunable durability up to exactly-once.
