OAuth 2.0

Jun 17, 2023 · 11 min read · Tech Stacks

OAuth 2.0 is an authorization framework that allows third-party applications to obtain limited access to a user's resources on a web service, without needing to know the user's login credentials. Before OAuth, the only way to give a third-party app access to your data was to hand over your username and password — a serious security risk. OAuth solves this by introducing a delegated-access model built around scoped, expiring tokens.

⚡ Quick Takeaways
  • OAuth 2.0 is authorization, not authentication — it says what you can do, not who you are. Add OpenID Connect (OIDC) on top for identity (the ID token).
  • Authorization Code + PKCE is the secure default for all user-facing flows (web and mobile); the authorization code is exchanged server-side, keeping tokens out of the browser.
  • Client Credentials for machine-to-machine — no user involved; service authenticates as itself with client_id + client_secret.
  • Implicit grant is deprecated — it returns tokens in the URL fragment, exposing them to browser history and JavaScript. Use Authorization Code + PKCE instead.
  • Access tokens should be short-lived (minutes to an hour); refresh tokens must be stored server-side and rotated on each use.
  • Always validate the state parameter in the Authorization Code flow to prevent CSRF attacks.
tldr

OAuth 2.0 is authorization (what you can do), not authentication (who you are). It introduces an authorization layer: instead of sharing credentials, the resource owner grants a client a scoped, expiring access token issued by the authorization server. Authorization Code flow is the secure default for server-side apps; Client Credentials is used for machine-to-machine.

Key Roles

RoleResponsibility
Resource OwnerThe end-user who owns the protected resource and grants permission for a client to access it.
ClientThe application requesting access (web app, mobile app, CLI tool, backend service).
Authorization ServerAuthenticates the resource owner, obtains consent, and issues access tokens and refresh tokens.
Resource ServerHosts the protected resources (e.g., a REST API). Validates access tokens on every request before serving data.
Access TokenA credential representing the granted authorization. Contains scope and expiration. Opaque strings or JWTs.

OAuth 2.0 vs. Authentication

A common interview mistake: confusing OAuth with authentication. OAuth 2.0 is purely about authorization — granting a client permission to perform actions on behalf of a user. It does not tell you who the user is. OpenID Connect (OIDC) is the authentication layer built on top of OAuth 2.0 — it adds an ID token (a JWT) that carries identity claims like name and email. When you click "Login with Google," you are using OIDC (identity) over OAuth (authorization).

Authorization Grant Types

Authorization Code (Recommended for Server-Side Apps)

The most secure flow. The client never sees the user's credentials. A short-lived authorization code is exchanged server-to-server (with the client secret) for tokens, keeping the actual tokens out of the browser.

  1. User clicks "Login with X" — client redirects user to the authorization server with response_type=code, scope, and a state parameter (CSRF protection).
  2. User authenticates and consents — authorization server redirects back to the client's redirect_uri with a short-lived code.
  3. Client's backend exchanges the code for tokens — POST to the token endpoint with code, client_id, and client_secret.
  4. Authorization server returns access_token and optionally refresh_token.
http 
# Step 1 — redirect user to auth server
GET https://auth.example.com/authorize
  ?response_type=code
  &client_id=my-app
  &redirect_uri=https://myapp.com/callback
  &scope=read:profile email
  &state=random-csrf-token

# Step 3 — exchange code for tokens (server-side)
POST https://auth.example.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=SplxlOBeZQQYbYS6WxSbIA
&redirect_uri=https://myapp.com/callback
&client_id=my-app
&client_secret=my-secret

Client Credentials (Machine-to-Machine)

Used when there is no human user involved — a backend service authenticating directly as itself. The client sends its client_id and client_secret to the token endpoint and receives an access token. Common for microservice-to-microservice calls, cron jobs, and CI pipelines accessing APIs.

Implicit (Deprecated)

Designed for single-page apps before PKCE existed: the token is returned directly in the URL fragment, skipping the code exchange step. This exposes the token to browser history, referrer headers, and any JavaScript running on the page. Do not use for new applications. Replace with Authorization Code + PKCE.

Resource Owner Password Credentials (Avoid)

The client collects the user's username and password directly and sends them to the authorization server. Defeats the purpose of OAuth — the user has to trust the client with credentials. Only appropriate for highly trusted first-party clients during a legacy migration.

Refresh Token

Access tokens are short-lived (typically minutes to an hour) to limit the damage of token theft. A refresh token is a long-lived credential that the client can use to obtain a new access token without prompting the user again. Refresh tokens must be stored securely (server-side, never in localStorage) and should be rotated on each use.

PKCE (Proof Key for Code Exchange)

PKCE (pronounced "pixie") is an extension to the Authorization Code flow designed for public clients — mobile apps and SPAs that cannot safely store a client secret. The client generates a random code_verifier, hashes it into a code_challenge, and includes the challenge in the authorization request. When exchanging the code for a token, it sends the original code_verifier. The authorization server verifies the hash, proving the token request came from the same party that initiated the auth request — preventing authorization code interception attacks.

Use Cases

  • Third-party app integration — "Connect your Google Calendar" — the app gets read access to calendar data without ever seeing your Google password.
  • Single Sign-On (SSO) — authenticate once with an identity provider (Okta, Auth0, Google Workspace) and access multiple internal applications without re-logging in.
  • API authorization — issue scoped tokens to API consumers so they can only access the endpoints they need (scope=read:orders), not your entire API surface.
  • Federated identity — multiple services delegate identity management to a central identity provider, reducing each service's attack surface and credential management overhead.
takeaway

Use Authorization Code + PKCE for all user-facing flows — web and mobile alike. Use Client Credentials for service-to-service. Never use Implicit or Password grant for new systems. Keep access tokens short-lived, rotate refresh tokens, and always validate the state parameter to prevent CSRF. Remember: OAuth 2.0 is authorization — layer OpenID Connect on top if you need identity.

🎯 interview hot-takes

OAuth vs OpenID Connect — what's the difference? OAuth 2.0 is authorization (can this app access your calendar?). OIDC adds authentication on top — it issues an ID token (JWT) with identity claims like name and email. "Login with Google" uses OIDC over OAuth.
Why is the Implicit grant deprecated? Tokens are returned in the URL fragment, exposing them to browser history, referrer headers, and any JavaScript on the page. Authorization Code + PKCE achieves the same goal for SPAs without this exposure.
What does PKCE protect against? Authorization code interception attacks. The code_challenge proves the token exchange came from the same party that initiated the auth request, even without a client secret.
