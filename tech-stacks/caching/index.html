Caching

Apr 02, 2023 · 11 min read · Tech Stacks

Caching stores the result of an expensive operation — a database query, an API call, a computed value — in fast storage so subsequent requests can be served without repeating the work. It is one of the highest-leverage performance techniques in distributed systems: a well-placed cache can reduce database load by orders of magnitude and cut response times from hundreds of milliseconds to single digits.

⚡ Quick Takeaways
  • Cache-aside (lazy loading) is the safest default — only cache what's actually requested, and degrade gracefully on cache failure.
  • Write-through keeps cache and DB in sync but wastes space on data that's never read; write-behind maximizes write throughput but risks data loss on crash.
  • LRU is the general-purpose eviction policy; use LFU for stable hot-key workloads; add TTL jitter to prevent avalanche.
  • Cache stampede — when a popular key expires, many threads hit the DB simultaneously; fix with mutex locking or probabilistic early refresh.
  • Redis vs Memcached — Redis wins for rich data structures, persistence, and pub/sub; Memcached wins for pure simple string throughput at scale.
  • Bloom filter prevents cache penetration for non-existent keys without a DB lookup.
tldr

Cache frequently-read, rarely-changing data close to the consumer. Choose a write strategy (cache-aside, write-through, write-behind) based on your consistency needs. Use TTL and an eviction policy (LRU is the safe default) to bound memory. Design for cache misses — a cold start or stampede should degrade gracefully, not cascade into a database meltdown.

Where to Cache

Caches can live at multiple layers of the stack, each with different tradeoffs:

  • Client-side (browser) — HTTP caching headers (Cache-Control, ETag) let browsers reuse assets without any server round-trip. Free performance for static content.
  • CDN (edge cache) — Geographically distributed caches serve static and semi-static content from points of presence close to users. Reduces latency and origin server load.
  • Application-level in-process — A simple hash map or a library like Caffeine (JVM) lives inside the app process. Zero network overhead, but not shared across instances and lost on restart.
  • Distributed cache — A shared external store (Redis, Memcached) accessible by all application instances. The standard approach for session data, computed results, and hot database rows.

Cache Write Strategies

Cache-Aside (Lazy Loading)

The application owns the cache interaction. On a read: check the cache first; if it misses, query the database, write the result to the cache, and return it. Writes go directly to the database; the cache entry is invalidated or left to expire.

  • Pro: Only requested data is cached — no wasted memory on cold data.
  • Con: First request after a miss is slow (cache cold). Stale data can accumulate if invalidation is missed.

Write-Through

Every write goes to both the cache and the database synchronously before acknowledging success. The cache is always consistent with the database.

  • Pro: Strong consistency — reads always hit a warm, accurate cache.
  • Con: Every write pays the cache write penalty. Data that is never read wastes cache space.

Write-Behind (Write-Back)

Writes land in the cache immediately; the database write is deferred and batched asynchronously. Maximizes write throughput.

  • Pro: Very fast writes; reduces database write pressure.
  • Con: Risk of data loss if the cache node fails before flushing to the database. Complex failure handling.

Eviction Policies

When the cache is full, something must be evicted to make room. The right policy depends on your access pattern:

PolicyEvictsBest For
LRU (Least Recently Used)The item not accessed for the longest timeGeneral-purpose; most access patterns have temporal locality
LFU (Least Frequently Used)The item with the fewest accessesWorkloads with stable hot keys (viral content, top-N lists)
FIFOThe oldest-inserted itemSimple queues; rarely optimal for caches
TTL (Time To Live)Items past their expiry timestampData with a known freshness window (rate limit counters, sessions)

Redis vs. Memcached

DimensionRedisMemcached
Data structuresStrings, hashes, lists, sets, sorted sets, streams, bitmapsStrings only
PersistenceRDB snapshots + AOF logNone (memory only)
ClusteringBuilt-in cluster mode, Sentinel for HAClient-side consistent hashing
Pub/SubYesNo
Use caseSessions, leaderboards, rate limiting, queuesSimple key-value caching, high-throughput reads

Redis is the default choice for most new systems. Memcached retains an edge in pure throughput for simple string caching on multi-core machines because its threading model is simpler, but that gap has narrowed significantly with Redis 6+ multi-threading.

Common Failure Modes

Cache Stampede (Thundering Herd)

When a popular key expires, many requests simultaneously miss the cache and all hit the database at once. Solutions: probabilistic early expiration (refresh slightly before TTL with some probability), mutex locking (first miss acquires a lock and populates the cache; others wait), or staggered TTLs (add random jitter to expiration times).

Cache Penetration

Requests for keys that do not exist in either the cache or the database pass through to the database on every call. A common cause is malicious or buggy clients querying invalid IDs. Fix: cache negative results (a sentinel value like null with a short TTL), or use a Bloom filter in front of the cache to reject clearly invalid keys without a cache or database lookup.

Cache Avalanche

A large batch of keys expires simultaneously, sending a flood of cache misses to the database at the same time. Prevent it by spreading TTLs with random jitter on write, pre-warming the cache before rolling deployments, and setting a circuit breaker on the database connection pool.

takeaway

Cache-aside is the safest starting point — it is simple, only caches what is actually needed, and degrades gracefully on cache failure. Add TTL jitter from day one to avoid avalanche. Reach for Redis when you need persistence, pub/sub, or rich data structures; Memcached when you need the absolute simplest key-value store with minimal operational overhead.

🎯 interview hot-takes

Cache stampede vs cache avalanche — what's the difference? Stampede: one popular key expires and many threads hit the DB at once. Avalanche: many keys expire simultaneously sending a flood of misses. Both are solved with TTL jitter; stampede additionally needs mutex locking or early probabilistic refresh.
How do you handle cache penetration? Cache negative results (a sentinel like "__null__" with a short TTL) or place a Bloom filter in front to reject invalid keys before hitting the cache or DB.
Redis or Memcached? Redis for anything beyond simple string caching — it supports persistence, pub/sub, sorted sets for leaderboards, and streams. Memcached only when you need the absolute highest throughput for plain string lookups with minimal ops overhead.
