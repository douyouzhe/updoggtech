Message Queue

Apr 02, 2023 · 10 min read · Tech Stacks

A message queue is a durable buffer that decouples the producer of a message from its consumer. The producer sends a message to the queue and moves on; one or more consumers process it asynchronously, at their own pace. This asynchronous handoff is what makes message queues a fundamental building block for resilient, loosely-coupled distributed systems.

tldr

Message queues decouple producers from consumers in time and speed. They absorb traffic bursts, enable async work, and improve system resilience. Choose between at-least-once and exactly-once delivery based on whether your consumers are idempotent. Use dead-letter queues to handle poison messages without blocking the main queue.

Core Problems Message Queues Solve

  • Traffic spikes — a queue absorbs burst traffic so downstream services are not overwhelmed. The producer writes at its pace; the consumer processes at its pace. This is sometimes called the "shock absorber" pattern.
  • Service decoupling — the producer does not need to know about the consumer, its location, or whether it is currently running. Each service evolves independently.
  • Async offloading — work that does not need to block the user response (sending email, resizing images, generating reports) can be queued and handled out-of-band, reducing user-facing latency.
  • Resilience — if a downstream service crashes, messages accumulate in the queue. When the service recovers, it processes the backlog without any message loss.

Delivery Semantics

SemanticGuaranteeRiskWhen to Use
At-Most-OnceMessage delivered zero or one timeMessage lossMetrics, heartbeats — loss is tolerable
At-Least-OnceMessage delivered one or more timesDuplicatesMost use cases — requires idempotent consumers
Exactly-OnceMessage delivered exactly one timeHigh complexity / latencyFinancial transactions, inventory updates

At-least-once is the practical default. The burden of handling duplicates is placed on the consumer — typically by checking a deduplication ID before processing, or by making the operation naturally idempotent (e.g., SET user.status = 'active' is safe to run twice; balance -= 100 is not).

Push vs. Pull

Push — the broker delivers messages to consumers as they arrive. Low latency, but the broker must track consumer capacity to avoid overwhelming a slow consumer.

Pull — consumers poll the broker for new messages. This naturally rate-limits delivery to the consumer's capacity and is the model used by Kafka and SQS. The downside is that polling adds a small latency floor and wastes CPU with empty-poll cycles, which is mitigated by long polling (the broker holds the connection open until a message arrives or a timeout expires).

Dead-Letter Queues (DLQ)

When a consumer repeatedly fails to process a message — due to a malformed payload, a transient bug, or an incompatible schema change — that message becomes a "poison pill" that blocks the queue. A dead-letter queue is a secondary queue to which messages are moved after exceeding a maximum delivery attempt count (maxReceiveCount in SQS). This keeps the main queue flowing while preserving the failed message for inspection and replay.

json 
// AWS SQS: attach a DLQ to a source queue
{
  "deadLetterTargetArn": "arn:aws:sqs:us-east-1:123456:my-dlq",
  "maxReceiveCount": 5   // move to DLQ after 5 failed deliveries
}

DimensionTraditional MQ (RabbitMQ, SQS)Kafka
Message lifecycleDeleted after successful consumptionRetained for a configurable period; multiple consumers can re-read
Consumer modelCompeting consumers — one consumer per messageConsumer groups — multiple independent groups each read all messages
OrderingFIFO within a queue (with caveats)Strict ordering within a partition
ThroughputHigh, but lower ceilingVery high — designed for streaming at scale
Best forTask queues, job distribution, RPCEvent streaming, audit logs, data pipelines

Idempotency in Consumers

Because at-least-once delivery is the norm, well-designed consumers must be idempotent — processing the same message twice produces the same result as processing it once. Common patterns:

  • Idempotency key — include a unique message ID in every message. The consumer checks a processed-IDs store before acting; skip if already seen.
  • Natural idempotency — design operations so duplication is harmless: upserts instead of inserts, SET instead of INCREMENT.
  • Database transactions — wrap the consume + state-update in a single transaction with a uniqueness constraint on the message ID column.
takeaway

Use message queues to decouple services in time and protect downstream systems from traffic spikes. Design consumers to be idempotent by default — at-least-once delivery is the realistic guarantee in any distributed system. Wire up a dead-letter queue from day one so poison messages surface immediately rather than silently blocking your processing pipeline.
