System Design

Design a Chat App

Apr 01, 2023 · 11 min read · System Design

A chat application sits at one of the most demanding intersections of system design: it must handle billions of users, hundreds of billions of messages per day, sub-second delivery latency, reliable offline queuing, and complex group semantics — all while maintaining end-to-end encryption and accurate presence information.

tldr

Use WebSockets for full-duplex, persistent client-server connections. Route messages through a WebSocket Manager (key-value store) that maps user IDs to servers. Generate message IDs with Snowflake (time-ordered, distributed, 64-bit). Queue messages as "pending" for offline users and deliver via push notification. Cap group size to bound fan-out cost.

Requirements

Functional

  • User registration and authentication
  • One-to-one messaging
  • Group chat
  • Media support: text, images, audio, video
  • Read receipts
  • Last seen / online status tracking

Non-Functional

  • Low latency — real-time feel is the core product promise
  • High availability — dropped connections or missed messages destroy trust
  • Massive scale — 4 billion users, 100 billion messages daily

Protocol Selection: Why WebSockets

The central challenge of chat is server-to-client message push. Three approaches exist:

ProtocolProblem
Short pollingLatency determined by poll interval; frequent polls waste resources, infrequent polls feel laggy
Long pollingRepeated connection setup/teardown overhead; multiple simultaneous requests can cause message ordering issues
WebSocketsPersistent full-duplex connection; eliminates polling overhead; supports true real-time bidirectional messaging

WebSockets win because they maintain a single persistent connection per client, enabling the server to push messages the instant they arrive with no polling lag and no connection re-establishment overhead.

Core Services

  • User Service — RESTful API for user CRUD and authentication
  • Chat / Group Service — manages chat creation, deletion, and group membership; stores [group_id, member_list] pairs
  • Message Service — persists messages and publishes them to Kafka for downstream processing
  • WebSocket Manager — maintains a key-value mapping of [user_id → websocket_server:port]; backed by Redis for fast in-memory lookup across 65,000+ available ports per server
  • Last Seen Service — samples user activity at ~1-minute intervals; records a timestamp when the app is open

Message Schema and ID Generation

schema 
-- 1-to-1 message
message_id      BIGINT   -- Snowflake ID (time-ordered)
sender_id       BIGINT
receiver_id     BIGINT
content         TEXT     -- encrypted
status          ENUM     -- Pending | Delivered | Read
created_at      TIMESTAMP

-- Group message (receiver_id replaced by channel)
message_id      BIGINT
sender_id       BIGINT
channel_id      BIGINT   -- broadcast to all members
content         TEXT
status          ENUM
created_at      TIMESTAMP

Message IDs use Snowflake: a 64-bit integer composed of a 41-bit millisecond timestamp, a 10-bit machine ID, and a 12-bit sequence number. This gives distributed uniqueness without coordination, and the timestamp prefix keeps messages naturally time-ordered for display.

Delivery: Online vs. Offline Users

When a message is sent, the Message Service checks whether the recipient has an active WebSocket connection by querying the WebSocket Manager. Two paths diverge:

  • Online user — message is routed directly to the recipient's WebSocket server and delivered immediately; status advances to Delivered
  • Offline user — message is stored with status Pending; a push notification is dispatched via Apple Push Notification Service or Android push notifiers; delivery is confirmed when the user reconnects

Group Chat and Fan-out

Group messages replace receiver_id with a channel_id and must be broadcast to all group members simultaneously. This fan-out is the main scalability challenge in group chat: a message to a 500-person group requires 500 individual deliveries. The design caps group size (WeChat's 500-user ceiling is cited as the reference) to bound the maximum fan-out cost and reduce delivery lag. The system accepts slightly different delivery timing across group members as a reasonable tradeoff for performance.

Media Storage

Images, audio, and video are not stored inline in message objects. Instead, the message record holds a reference, and the actual binary content lives in a file system backed by CDN distribution. This keeps message database rows small and uniform while leveraging optimized object storage for large binary payloads.

Last Seen and Presence

Recording exact timestamps for every user action at 4 billion-user scale would generate enormous write load. Instead, the Last Seen Service samples: it polls whether the user has the app open approximately every minute and records a timestamp if so. This introduces minor inaccuracy in "last seen" timestamps — a deliberately accepted tradeoff to keep storage and write throughput manageable.

takeaway

Chat design is mostly about two hard problems: real-time delivery (WebSockets + WebSocket Manager for routing) and offline queuing (Pending status + push notifications). Snowflake IDs solve distributed unique ID generation with built-in time ordering. Group size caps are an explicit scalability lever — the smaller the cap, the cheaper the fan-out. Last seen sampling trades minor accuracy for write throughput.
