System Design

Design a Map Web App

Apr 01, 2023 · 11 min read · System Design

A map platform like Google Maps serves billions of users across wildly different use cases — turn-by-turn navigation, business discovery, satellite imagery, traffic-aware routing, and third-party integrations for apps like Uber. At 3 billion monthly active users and 10 million API partners, it is one of the most data-intensive and latency-sensitive systems in existence.

tldr

Pre-compute routes between sectional nodes and store them in a geohash-indexed key-value store (BigTable/DynamoDB/Cassandra). Use WebSockets for real-time navigation. Feed user location pings (every 5s, at-most-once) into a message queue. Let CDC-driven Kafka keep the Elasticsearch search index current with map changes. Push ML-powered ETA refinement and hotspot prediction offline.

Requirements

Functional

  • Identify roads, places, and routes on a map
  • Calculate accurate distance and ETA accounting for traffic, road conditions, and real-time incidents
  • Provide a pluggable API for third-party integration (ride-share apps, vehicle manufacturers, data providers)

Non-Functional

  • Low latency — route queries must return in milliseconds, not seconds
  • High availability — navigation failures mid-trip are critical user experiences
  • Massive scale — 3 billion monthly active users, 10 million API partners

Map Data: Ingestion and Storage

Initial map data is sourced from government municipal maps (e.g., USGS), satellite imagery providers (Planet, Maxar), street-level imagery vehicles, and third-party geographic data providers. Continuous updates arrive via user-submitted feedback, business submissions, and ongoing satellite refreshes.

All map modifications are processed through Kafka via Change Data Capture (CDC), enabling downstream consumers — search indexers, route pre-computers, analytics services — to react to map changes asynchronously.

Spatial Indexing with Geohash

The map is represented as latitude/longitude coordinates (a 2D x-y plane). To make spatial queries efficient at database scale, the system uses two complementary structures:

  • Quadtree — recursively subdivides regions into four quadrants, forming a hierarchical spatial index
  • Geohash — encodes 2D geographic coordinates into a 1D sortable string, enabling efficient range queries on a standard key-value store

The primary database of choice is Google's in-house single-keyed BigTable; at similar scale, AWS DynamoDB or Cassandra are viable alternatives for their massive data volume handling.

Routing Algorithm

Standard graph traversal (BFS) does not work for routing because road segments have variable weights (distance, speed limit, traffic). The correct algorithms are Dijkstra or Floyd-Warshall, which handle weighted edges. However, running Dijkstra across the full global road graph in real time has time complexity of O((V+E)logV) — far too expensive for 3 billion concurrent users.

Pre-Computation Strategy

The solution is to sacrifice storage for speed: routes between sampled nodes within each geographic section are pre-computed and stored. Navigation then resolves to a lookup across sections rather than a live graph traversal. Only the final unmapped micro-segment requires on-the-fly computation.

When a traffic incident causes a significant ETA change, the impact bubbles up recursively through the section hierarchy, updating only the affected route segments rather than recomputing globally.

Real-Time Navigation

Navigation sessions require bidirectional, low-latency communication between server and client — WebSockets are the right protocol here. A WebSocket Manager maintains a key-value mapping of connected users to their WebSocket server and port, enabling the Navigation Tracking Service to reach any active session.

Three events trigger a navigation update:

  • Route deviation — user strays from the planned path; recalculate
  • Significant ETA delay — a new incident on the route exceeds a threshold; notify user
  • Better route available — proactively reroute if a faster path opens up

User Location Tracking

Active users transmit their GPS location every 5 seconds (when location sharing is enabled). These packets flow into a message queue that decouples producers from consumers. The delivery guarantee is at-most-once: a single lost location packet is acceptable because the next one arrives 5 seconds later, and the cost of guaranteed delivery would outweigh the benefit for this use case.

Search and Discovery

Place name search is powered by an Elasticsearch cluster supporting fuzzy matching and type-ahead. The index is populated from two streams: direct place name data and map CDC events. Map browsing displays nodes with priority-based filtering that adapts to zoom level — zooming out shows only major landmarks, zooming in reveals fine-grained detail.

ML and Analysis Service

The Analysis Service consumes location and behavioral data to power several advanced features:

  • User profiling — driving habits, typical speed patterns, home/office location prediction
  • ETA refinement — percentile-based accuracy ranges using aggregated historical trip data
  • Proactive trip planning — departure reminders tied to calendar appointments or flight times
  • Data source scoring — reliability weighting for user-submitted incidents vs. third-party providers
  • Road discovery — identifying unmapped roads through aggregate user travel patterns
  • Hotspot prediction — forecasting congestion zones and surfacing alternative routes before congestion hits
takeaway

Map systems live or die on their spatial indexing and pre-computation strategy. Geohash bridges 2D geography to 1D key-value lookups; pre-computed routes bridge the gap between Dijkstra's correctness and real-time latency requirements. Everything else — CDC pipelines, WebSocket navigation, at-most-once location pings — is about keeping these pre-computed answers current and delivering them fast.
