Design a Hotel Booking App

Apr 01, 2023 · 10 min read · System Design

A hotel booking platform is a two-sided marketplace with strict correctness requirements on the critical path: double-booking a room is a severe UX failure, while incorrectly blocking available inventory costs revenue. The core design challenge is maintaining accurate room availability under concurrent bookings across a global user base.

tldr

Use SQL throughout for ACID correctness — hotel and room data is bounded in size, and bookings demand transaction integrity. A row-level constraint on available_count prevents overselling at the database level. A Redis hold with ~10-minute TTL bridges the checkout gap. Kafka feeds listing changes into Elasticsearch for search. CDN distributes hotel media globally.

Requirements

Functional (Hotel Admin)

  • CRUD operations for hotels and rooms
  • Manage bookings (view, cancel, modify)
  • Access business insights and reporting

Functional (User)

  • Search for hotels and available rooms by location, dates, and filters
  • Make reservations

Non-Functional

  • High consistency — no double-bookings; availability counts must be correct
  • Low latency — search and checkout must feel instant
  • High availability — a down booking service means lost revenue
  • Scalability — handles global user load and seasonal peaks (holidays, events)

Hotel and Room Listing Service

The hotel and room catalog is stored in a SQL database. The rationale is deliberately simple: the number of hotels in the world is finite and grows predictably. A relational database is the right fit — the data is structured, growth is foreseeable, and the query patterns (look up hotel by ID, list rooms by hotel) are well-suited to SQL indexes.

Hotel images and videos are stored in AWS S3 (or equivalent object storage). The SQL database holds references (URLs) to these media assets. A CDN sits in front of S3 to serve media to a global user base with low latency — fetching a hotel image from a geographically proximate CDN edge is orders of magnitude faster than hitting the origin bucket.

When hotels or rooms are created or updated, the changes are published to Kafka as events. A consumer reads these events and updates the Elasticsearch search index asynchronously, keeping search results eventually consistent with the source of truth.

Key APIs

REST API 
POST   /hotels              -- create hotel
PUT    /hotels/{id}         -- update hotel
GET    /hotels/{id}         -- get hotel details

PUT    /hotels/{id}/rooms/{room_id}   -- update room
GET    /hotels/{id}/rooms             -- list rooms for hotel

Search Service

Search is powered by an Elasticsearch cluster (Solr is a comparable alternative — both are built on Apache Lucene). The two differ in their strengths:

FeatureElasticsearchSolr
Best forTime-series data, real-time indexingStatic data sets with heavy caching
Fuzzy searchExcellentGood
Type-aheadBuilt-in completion suggesterSupported via edge n-gram

For a hotel platform where listings update continuously (price changes, availability, new properties), Elasticsearch is the better fit. The search index is updated via Kafka events whenever listings change, removing fully-booked rooms from search results automatically.

Booking Service and Availability Tracking

The booking service uses a SQL database for its ACID guarantees — a double-booking is unacceptable, and ACID transactions are the clean way to prevent it. Room availability is tracked in a dedicated table:

SQL schema 
CREATE TABLE room_availability (
  room_id         BIGINT,
  date            DATE,
  available_count INT CHECK (available_count >= 0),
  PRIMARY KEY (room_id, date)
);

-- Row-level constraint prevents available_count going negative
-- A booking that would set count to -1 raises a constraint violation

The CHECK (available_count >= 0) constraint is the database-level guard against overselling. When a concurrent transaction attempts to decrement below zero, it fails — enforcing correctness without application-level locking.

Preventing Overselling During Checkout

Between when a user selects a room and when they complete payment, other users might book the same room. The solution mirrors the e-commerce cart pattern: a Redis hold with a ~10-minute TTL soft-reserves the room during checkout. The flow:

  • User selects room → Redis hold created, available_count decremented tentatively
  • User completes payment → booking record written to SQL with status Booked; hold confirmed
  • User abandons checkout → Redis TTL expires → available_count automatically restored via cache invalidation
  • Payment fails → booking status set to Payment Declined; count restored

Booking Status Lifecycle

  • Reserved — Redis hold active; checkout in progress
  • Booked — payment confirmed; room locked
  • Payment Declined — payment failed; room released
  • User Cancelled — user cancelled confirmed booking
  • Completed — stay has occurred

Analytics and Reporting

The Analytics Service consumes two event streams from Kafka: search actions (what users search for, which results they click) and order transactions (bookings, cancellations, revenue). These feeds power:

  • Price optimization — surfacing demand signals to hotel managers for dynamic pricing
  • Business insights — occupancy rates, revenue trends, booking lead times
  • Accounting reports — reconciliation data for hotel finance teams

Why Not Cassandra for Bookings?

Unlike e-commerce order history (high volume, append-heavy, suitable for Cassandra archival), hotel booking volume is low enough that a SQL master database can retain the full history without a dedicated archival service. The correctness benefits of ACID transactions outweigh the scalability benefits of NoSQL at this data volume.

takeaway

Hotel booking design is all about consistency: SQL + row-level constraints prevent double-booking at the database level; Redis TTL holds solve the checkout gap without permanently blocking inventory; Kafka feeds Elasticsearch asynchronously to keep search accurate. The "why not NoSQL" answer here is simple — booking volume doesn't justify sacrificing ACID, and hotel counts are bounded.
