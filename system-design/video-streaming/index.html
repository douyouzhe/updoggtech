System Design

Design a Video Streaming Platform

Apr 01, 2023 · 11 min read · System Design

A video streaming platform like YouTube or Netflix must handle massive storage (hundreds of terabytes of daily uploads), complex multi-format transcoding, global CDN delivery with minimal buffering, and ML-driven recommendations that keep users engaged. The upload pipeline and the playback pipeline are effectively two separate systems with very different constraints.

tldr

Video uploads flow through a Kafka-backed pipeline of content filtering, classification, and multi-resolution transcoding before landing in blob storage and Cassandra metadata. Playback is served from geographically distributed CDNs. Recommendations are computed offline and stored in key-value NoSQL. TCP is chosen over UDP for reliability — no packet loss means no buffering artifacts.

Requirements

Functional

  • User authentication (register and login)
  • Video upload
  • Video playback
  • Homepage with content discovery
  • Multi-platform support: devices, formats, resolutions, quality levels, regions, and languages

Non-Functional

  • Low latency, high availability — minimal buffering is the primary user experience metric
  • User engagement — high-quality recommendation content keeps watch time high

Scale Estimates

  • 10 million daily active users, each watching ~10 videos per day
  • 1% are content creators uploading ~2 videos per day each
  • Average video size: 500 MB → approximately 100 TB of new storage daily

Protocol: TCP vs. UDP

YouTube and Netflix adopted TCP as the transport protocol primarily to guarantee video quality — TCP's retransmission ensures zero packet loss, eliminating the pixelation or blocking artifacts that dropped UDP packets cause. Security is a secondary benefit. The tradeoff is slightly higher latency compared to UDP, which is acceptable for on-demand streaming. UDP remains dominant in live streaming where low latency matters more than perfect quality. Google's QUIC protocol is an emerging alternative that achieves TCP-like reliability over UDP.

Upload Pipeline

The upload path is the most complex flow in the system. A raw video file goes through multiple transformations before it is ready for viewers:

flow 
Creator uploads raw video
  → Content Onboarding Initializer Service
  → Kafka message queue
      ├── Multi-part Resolver: chunks file for parallel processing
      ├── Content Filter: screens for illegal/inappropriate material
      ├── Classifier & Tagger: generates tags, ratings, categories
      └── Transcoder: produces multiple resolutions, formats, bitrates
            └── Blob storage (S3)
                └── Elasticsearch indexing
                    └── Content Service → Cassandra (metadata)

Chunking the upload into parallel parts through the Multi-part Resolver dramatically reduces total processing time for large files. Each chunk can be processed independently and re-uploaded on failure without restarting the entire transfer.

Storage Architecture

User Authentication

The authentication flow uses a standard RESTful API service backed by a SQL database for ACID guarantees. A Redis cache with Cache-Aside (lazy loading) pattern sits in front of the SQL database: data is loaded into cache on first miss and served from cache on subsequent reads. Lazy loading is preferred over write-through here because it avoids caching data that is never actually requested.

Content Delivery

Video files are served through a globally distributed CDN network. A Host Optimizer service directs each client to the nearest CDN node based on geographic proximity and current server load. Lazy loading is applied at the CDN layer — only the segments the user is currently watching are fetched, rather than the full video file, reducing bandwidth costs and startup latency.

Homepage and Search

The homepage combines two data sources:

  • Search Service — an Elasticsearch cluster with fuzzy search and type-ahead for active queries
  • Recommendations — pre-computed ML predictions stored in a key-value NoSQL database, keyed by user ID for O(1) lookup

Recommendations are computed offline from watch history, watch duration, and interaction signals (likes, shares, re-watches). These signals are fed into Kafka from the playback service and used to train recommendation models in batch.

Playback Tracking

During playback, the client continuously logs watch status and duration. This data flows into Kafka and drives two downstream consumers: the recommendation model training pipeline (offline) and the watch history service (near real-time). Continuous logging also enables resuming playback from the correct position across devices.

takeaway

Video streaming is two systems: upload (complex multi-stage pipeline optimized for throughput) and playback (optimized for latency via CDN and lazy loading). TCP's no-packet-loss guarantee is worth the latency tradeoff for on-demand streaming. Recommendations belong offline — pre-compute once, serve fast from key-value store, retrain continuously from Kafka-buffered behavior signals.
