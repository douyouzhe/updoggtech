Design a URL Shortener

Apr 01, 2023 · 9 min read · System Design

A URL shortener is a deceptively simple service: take a long URL, return a short one, and redirect the short one back to the original. The design challenge lies in generating short IDs that are unique at scale, handling collisions, keeping redirect latency near zero, and purging expired links without impacting hot-path performance.

tldr

6-character base-62 slugs give sufficient namespace for a year's worth of URLs. Use ID generation (Snowflake or range assignment) over hashing to avoid collisions. Add a Redis Bloom filter to check for existing slugs before writing. Cache hot redirects in Redis. Purge expired URLs via TTL, scheduled scans, or lazy evaluation. Rate-limit write endpoints to prevent DDoS.

Requirements

Functional

  • Generate a short URL from a long URL input
  • Redirect a short URL back to the original long URL

Non-Functional

  • Very low latency — redirects happen on every link click; every millisecond counts
  • Very high availability — a down service breaks every shortened link across the internet

Scale Estimates

  • 10 million DAU, ~2 requests per user per day
  • Write QPS: ~230 (peak: ~460)
  • Read QPS: ~2,300 (10:1 read-to-write ratio)
  • Storage: less than 1 TB for a 1-year TTL (at ~100 bytes per entry)
  • Short URL length: 6 characters from a base-62 character set [0-9, a-z, A-Z] yields 62⁶ > 56 billion unique slugs — more than enough for annual URL volume

Short ID Generation: Hashing vs. ID Generation

The core question is how to generate the 6-character slug:

ApproachMechanismProblem
Hashing (MD5, SHA-1)Hash the long URL, take first 6 charsDifferent long URLs can produce the same 6-char prefix — collisions require detection and retry
UUIDGenerate random 128-bit ID, encode to base-62Low collision probability but requires entropy; IDs are not naturally sortable
Snowflake ID41-bit timestamp + 10-bit machine ID + 12-bit sequenceNo collision if machine IDs are unique; time-ordered; requires coordination for machine ID assignment
Range assignmentPartition a numeric range across servers; round-robin or contiguous blocksRequires an external controller to assign ranges; simpler than Snowflake for small clusters

Hashing requires a collision detection layer. The canonical solution is a Redis Bloom filter: before writing a new slug, check the Bloom filter. A negative result means the slug is definitely new (safe to write). A positive result means the slug might already exist — fall back to a database lookup. This keeps the fast path (new slug) extremely cheap.

scalability note

A single Redis Bloom filter becomes a bottleneck at high scale. Distributing Bloom filter checks across multiple Redis instances reduces the bottleneck but sacrifices a single source of truth — a known tradeoff without a clean solution at extreme scale.

Core Architecture

flow 
-- Write path (shorten a URL)
POST /shorten { long_url }
  → Rate limiter (by user token)
  → Generate slug (Snowflake / range / hash+salt)
  → Check Redis Bloom filter for collision
  → Write { slug → long_url, created_at, ttl } to DB
  → Return short URL

-- Read path (redirect)
GET /{slug}
  → Check Redis cache  (cache hit → 301/302 redirect)
  → On miss: DB lookup → cache → redirect
  → Log click event for analytics

Redirect: 301 vs. 302

The HTTP redirect response code matters for caching behavior:

  • 301 (Permanent Redirect) — browsers cache the destination; subsequent clicks skip the shortener entirely. Reduces load but loses click analytics after the first visit.
  • 302 (Temporary Redirect) — browsers do not cache; every click hits the shortener. Preserves analytics at the cost of higher server load.

The right choice depends on whether click-level analytics are a product requirement.

URL Expiration and Purging

Expired URLs must be cleaned up to reclaim storage and slug namespace. Three strategies exist:

  • Redis TTL — set an expiration on the cache entry; automatic cleanup with no database overhead. Works well when the cache is the primary read path.
  • Scheduled database scans — an offline Airflow job periodically scans for rows where created_at + ttl < now() and deletes them. Simple but potentially slow on large tables.
  • Lazy evaluation — check the TTL on every redirect request; return a 404 and delete if expired. Zero dedicated cleanup work, but expired rows accumulate until accessed.

Rate Limiting and Security

The write endpoint must be protected with a rate limiter to prevent DDoS attacks that generate millions of slugs, exhausting both storage and the slug namespace. User-token-based tracking enables per-user quotas. The read endpoint is generally left unthrottled since redirects are cheap and caching absorbs most load.

takeaway

URL shortener design is a study in making boring things scale: ID generation beats hashing (avoid collisions at the source), Bloom filters make collision checks cheap, Redis caching absorbs the 10:1 read amplification, and slug length arithmetic confirms 6 characters is sufficient. The subtle choices — 301 vs. 302, which purge strategy — are driven by product requirements, not technical constraints.
