System Design

Design an E-commerce Platform

Mar 31, 2023 · 12 min read · System Design

An e-commerce platform is one of the most demanding distributed systems to design. It combines real-time inventory management, financial transaction processing, high-traffic search and discovery, and ML-driven recommendations — all under strict consistency requirements where an oversold item or a lost order directly translates to lost revenue and user trust.

⚡ Quick Takeaways
  • Microservices + API gateway — each service owns its data store; no cross-service direct DB access.
  • SQL for money, NoSQL for catalogs — ACID for orders/users; document store for heterogeneous product schemas.
  • Redis cart hold (TTL ~10 min) — soft-reserves inventory without permanently blocking stock; auto-released on expiry.
  • Kafka decouples checkout — user gets instant acknowledgement; inventory deduction, payment, and notifications happen async downstream.
  • CDC → Elasticsearch — search index stays eventually consistent without coupling write latency to indexing.
  • Cassandra for archival — horizontally scalable, high write throughput for historical order records.
tldr

Use microservices behind an API gateway: SQL for orders and users (ACID), NoSQL for product catalogs (schema flexibility), Cassandra for historical order archival, Redis with TTL for cart holds, and Kafka to decouple the checkout pipeline from downstream processing. Elasticsearch powers search via CDC-fed indexing.

Requirements

Functional

  • User registration and authentication
  • Merchant shop and product management (CRUD)
  • Search, cart, and wishlist capabilities
  • Checkout and purchase completion
  • Homepage feed and order history viewing

Non-Functional

  • Low latency — product search and checkout must feel instant
  • High availability — downtime during peak sales (Prime Day, 11/11) is unacceptable
  • High consistency — inventory and payment state must be correct
  • Revenue maximization — the system should never over-restrict sales or oversell inventory

Core Services

The platform is decomposed into focused microservices, each owning its data store:

  • User Service — authentication, profile management
  • Shop Service — merchant store creation and settings
  • Product Service — catalog CRUD, attributes, media
  • Search Service — Elasticsearch-powered full-text and faceted search
  • Serviceability Service — filters products by purchase eligibility (region, availability)
  • Inventory Service — stock tracking and reservation
  • Pricing Service — dynamic pricing calculations and promotions
  • Order Service — transaction processing and order lifecycle
  • Recommendation Service — ML-driven personalized suggestions
  • Notification Service — email, SMS, and push updates

Data Storage Decisions

Storage selection is driven by each service's access patterns and consistency requirements:

Data TypeStorageRationale
Users, orders, shopsSQL (RDBMS)ACID properties required for financial transactions and structured relational data
Product catalogsNoSQL (Document)Products have heterogeneous attribute sets; document stores avoid null-heavy schemas
Historical ordersCassandraHorizontally scalable, high write throughput without read degradation; suitable for append-heavy archival
Cart holdsRedis (TTL)~10-minute TTL temporarily reserves inventory without permanently blocking stock from other buyers
Search indexElasticsearchFull-text search, fuzzy matching, and faceted filtering at scale

The Checkout Pipeline

Checkout is the most critical — and most complex — flow. The key design insight is to decouple the user-facing checkout confirmation from the downstream order processing chain using Kafka. The user gets an immediate acknowledgement; the inventory deduction, payment charge, notification dispatch, and fulfillment kick-off happen asynchronously.

flow 
User clicks "Buy"
  → Cart Service validates TTL hold still active
  → Order Service writes pending order to SQL
  → Publishes OrderCreated event to Kafka
      ├── Inventory Service: deduct stock
      ├── Payment Service (Stripe/PayPal): charge card
      │     └── Webhook callback updates order status
      ├── Notification Service: send confirmation email
      └── Recommendation Service: record purchase signal

External payment providers (Stripe, PayPal) integrate via webhooks that fire on payment success or failure, driving the order's status state machine.

Search and Discovery

Product and shop updates flow into the Elasticsearch cluster via Change Data Capture (CDC) — the primary databases emit change events that a CDC connector (e.g., Debezium) publishes to Kafka, which then feeds the search indexer. This keeps the search index eventually consistent without coupling write latency to indexing.

The Search Service handles fuzzy matching and type-ahead queries. The Serviceability Service filters results by the requesting user's region and purchase eligibility before returning results to the client.

Scaling for Peak Traffic

Flash sales and seasonal peaks (like 11/11) can spike traffic by orders of magnitude. Two processing models handle different merchant segments:

  • Large enterprises (batch pull) — Apache Airflow schedules bulk pulls; better for high-volume, scheduled inventory updates
  • SMEs (push model) — events are immediately distributed upon product or order changes

The async Kafka pipeline also serves as a natural buffer during these surges, preventing the order processing back-end from being overwhelmed by sudden checkout bursts.

ML and Recommendations

User behavior signals — search queries, cart additions, wishlist saves, and purchase completions — are captured via Kafka and fed into Spark Streaming for real-time analysis. The Recommendation Service uses these signals for:

  • User segmentation (regression models, KNN clustering)
  • Personalized product recommendations on the homepage
  • Dynamic pricing algorithms
  • Demand forecasting to assist inventory planning

Every database table includes created_at and updated_at timestamp columns, which serve double duty: debugging production issues and providing temporal features for ML models.

Inventory Consistency Tradeoff

The hardest consistency problem in e-commerce is the cart-to-purchase gap: if you deduct inventory when the item is added to cart, you block sales for users who abandon carts. If you deduct only at checkout, you risk overselling during high-demand bursts. The chosen middle ground is a Redis hold with TTL (~10 minutes): inventory is soft-reserved on cart add and permanently deducted on successful checkout. Expired holds are automatically released by Redis.

takeaway

E-commerce system design is fundamentally about isolating consistency domains: use SQL + ACID for money and orders, document stores for flexible catalogs, Cassandra for archival scale, and Kafka to decouple the checkout critical path from all downstream effects. The Redis TTL cart hold is the canonical answer to the oversell vs. lost-sale tradeoff.

🎯 interview hot-takes

How do you prevent overselling during flash sales? Redis TTL hold (~10 min) soft-reserves inventory on cart add; permanent deduction only on confirmed checkout; expired holds auto-release.
Why Kafka between checkout and payment? Decouples user-facing latency from downstream processing — user gets instant order confirmation while inventory, payment, and notifications proceed asynchronously.
Why CDC into Elasticsearch instead of dual-writes? CDC guarantees the search index stays consistent with the primary DB without coupling write latency to indexing; dual-writes risk divergence if the index write fails.
